大模型

  1. 指的是参数量极大（Billion+），模型本身显存和计算需求高
  2. 单卡放不下 / 单机训练时间太长
  3. 必须用分布式训练手段才能完成
  4. 典型例子：LLaMA3-8B、70B，GPT-4，Gemini 等

模型训练

包括正向传播、反向传播、梯度更新

  • 前向传播：根据当前参数计算模型输出和损失
  • 反向传播：用链式法则从损失往回，层层计算出各个参数的梯度
  • 参数更新：根据计算出的梯度，更新参数

分布式训练

  • 是为了解决大模型训练的计算 / 存储瓶颈
  • 拆分模型（TP）、拆分数据（DP）、拆分层（PP）、拆分状态（ZeRO / FSDP）
  • 需要分布式通信、同步、协调
  • 涉及并行策略 + 启动工具 + 通信后端 + 容错机制

分布式训练关键参数说明

  1. WORLD_SIZE
    总 GPU 数量，表示整个训练中参与的 GPU 数量。

  2. TP_SIZE（Tensor Parallel Size）
    Tensor 并行度：将单层模型的巨大参数拆分成多份，每个 GPU 计算一部分，多个 GPU 协作完成该层的前后向计算。

    • 减小单 GPU 显存压力
    • 支持更大模型

  3. PP_SIZE（Pipeline Parallel Size）
    Pipeline 并行度：将模型的不同层划分为多个“段”，每个 GPU 负责其中一段的前向和反向传播。

    • 进一步分摊显存
    • 支持更深更大的模型结构

  4. DP_SIZE（Data Parallel Size）
    数据并行度：经典的并行方式，每个 GPU 保持模型完整副本，处理不同的 batch 数据，并在每个 step 同步梯度。

    • DP_SIZE = WORLD_SIZE / (TP_SIZE × PP_SIZE)
    • 表示一个模型并行组的数量，每组对应一个数据并行副本

  5. MICRO_BATCH_SIZE
    微批大小：指单次前向 + 反向的 batch 大小。多个微批连续执行，累计梯度后再更新一次参数。

  6. NUM_MICROBATCHES
    一个 step 内，在梯度累计前连续运行的微批次数量。

  7. GLOBAL_BATCH_SIZE
    全局批大小：一次参数更新时模型“整体”看到的样本数，是训练稳定性和模型效果的重要因素。大模型常用大 batch，有助于加速收敛。

    • GLOBAL_BATCH_SIZE = DP_SIZE × NUM_MICROBATCHES × MICRO_BATCH_SIZE
    • 相当于小模型中一个 step 的 batch_size，只不过这里是分布式场景中整组模型并行副本累计处理的总样本数

torchrun 启动机制

用于启动多个分布式训练进程，配合 Megatron-LM 驱动大模型训练。

  • 自动为每个 GPU 启动一个 Python 进程（单机多卡）
  • 多节点时，协调每个节点与 master 建立通信
  • 初始化 torch.distributed 环境
  • 每个进程内使用 torch.distributed.get_rank() 获取全局 rank
  • 后续的 DDP、模型并行、AllReduce 等都依赖这个初始化

分布式常见参数说明

参数名 说明
world_size 总进程数 = nnodes × nproc_per_node
rank 每个进程的全局编号（0 ~ world_size-1）
local_rank 每个节点内的进程编号（0 ~ nproc_per_node-1）
master_addr & master_port 用于集群通信的协调节点

分布式通信机制

基本通信原语（PyTorch Distributed）

  • AllReduce：所有 GPU 的张量求和/平均 → 同步梯度
  • Broadcast：从 rank 0 广播数据到其他进程
  • AllGather / ReduceScatter：多张量收集与分发
  • Barrier：全体进程阻塞同步，常用于初始化或 barrier 控制点

通信算子与应用场景

通信算子 作用描述 应用场景
AllReduce 张量求和/平均，同步到所有 GPU 同步梯度（DDP）
Broadcast 从 rank 0 广播张量 初始化模型参数
AllGather 所有进程张量拼接 模型并行中拼接切片
ReduceScatter 先规约后切片 张量并行中中间通信
Barrier 阻塞等待所有进程 开始训练或加载 checkpoint

通信分组（以 Megatron-LM 为例）

  • 数据并行组：同步梯度
  • 张量并行组：分片权重计算中通信
  • 流水线并行组：跨 stage 的激活通信

通信后端

后端名称 说明
NCCL CUDA 最快，NVIDIA 优化，支持 NVLink / PCIe / IB
MCCL MUSA 的通信后端（国产 GPU）
Gloo 通用后端，适用于 CPU、调试

NCCL Ring AllReduce / AllGather / ReduceScatter 原理简述

  • AllReduce：全卡归约并广播，分布式梯度同步的核心算子。
  • Ring AllReduce：将计算/通信均匀分布到所有进程，无 master 瓶颈，通信效率高。
  • ReduceScatter：AllReduce 的前半部分，规约后切片分发。
  • AllGather：AllReduce 的后半部分，将切片拼接广播。
  • Broadcast / Barrier：点对多点广播，全体同步阻塞。

通信模块的四个层次

  1. 通信原语（逻辑层）：AllReduce、Broadcast 等
  2. 通信库（软件层）：NCCL、MPI、Gloo
  3. 协议与驱动（接口层）：RDMA、TCP、驱动程序
  4. 真实硬件（物理层）：GPU、PCIe、NVLink、IB、交换机

分布式并行策略

单机多卡策略

数据并行（Data Parallel）

  • 每卡持有模型完整副本，分批处理不同输入，梯度通过 AllReduce 同步
  • ✅ 简单易实现，支持成熟（如 PyTorch DDP）
  • ❌ 模型必须单卡可容纳；AllReduce 开销随卡数上升

张量并行（Tensor Parallel）

  • 将某一层的参数（如矩阵）按维度切分，分卡并行计算
  • 用于降低单卡显存
  • 通信较频繁，需要 AllGather、ReduceScatter
  • 应用：Megatron-LM 的 1D 分片

流水线并行（Pipeline Parallel）

  • 将模型划分成多个 Stage，多个微批次以流水线形式穿过各卡
  • 优点：大幅降低激活显存占用
  • 缺点：流水线填空/排空带来利用率下降，通信复杂

混合并行（Hybrid）

  • 常见为 2D 或 3D 并行：张量 + 流水线 + 数据并行组合
  • 框架支持：Megatron-LM、DeepSpeed（3D Parallel）

多机多卡策略

跨机数据并行

  • 每机内部数据并行，节点间通过 NCCL 做梯度 AllReduce
  • 关键：网络带宽、拓扑感知、NCCL 环境变量调优

跨机模型并行

  • 张量并行 / 流水线并行可跨机拓展
    • 跨机张量并行：tensor 切片分布于多机
    • 跨机流水线并行：stage 可跨机部署
    • 支持 3D / 4D 混合并行（如 DeepSpeed）

主流框架支持对比

技术 / 框架 支持的并行类型 说明
PyTorch DDP 数据并行 简单易用，入门推荐
Megatron‑LM 1D 张量 + 流水线 + 数据并行 超大模型训练常用
DeepSpeed ZeRO (1/2/3) + Tensor + Pipeline + Data 支持上千亿参数
FairScale FSDP + 张量并行 PyPI 轻量包，适合实验

从单机到多机训练的推荐步骤

  1. 单机开发调试

    • 先用数据并行，确保代码正确
    • 用小模型 + 小 batch 验证显存/性能

  2. 加入模型并行

    • 单卡显存不够 → 加入张量并行 / 流水线并行
    • 加入激活检查点，节省显存

  3. 扩展到多机

    • 配置 NCCL、InfiniBand、确认 GPU 拓扑
    • 设置 init_process_group()

  4. 性能调优

    • 通信：调 NCCL 环境变量
    • 流水线：设置 micro-batch 数量、stage 划分平衡通信
    • ZeRO：根据显存与带宽选 Stage1/2/3

  5. 监控与 Profiling

    • 显存：torch.cuda.memory_summary()
    • 时间：PyTorch Profiler、DeepSpeed Profiler
    • 通信：nvidia-smi topo --matrix

总结

类型 特点
数据并行 入门首选，易扩展，显存限制
模型并行 张量 + 流水线，缓解显存压力，复杂度提升
ZeRO 拆分优化器/梯度/参数，占用最低显存
混合并行 超大模型训练主流，3D 组合策略

PyTorch Profiler — PyTorch Tutorials 2.5.0+cu124 documentation

用于分析PyTorch程序的性能，帮助开者深入了解模型的运行情况，以便进行优化。它能够捕捉和记录PyTorch程序在训练和推理时的各项性能指标，包括时间消耗、内存使用、各个操作的执行情况等，帮助找出瓶颈并进行优化。

指标

GPU 利用率

GPU繁忙时间与所有步骤时间的比值，越高越好。所有步骤时间是所有分析器步骤的总时间。GPU繁忙时间是“所有步骤时间”期间至少有一个GPU内核在此GPU上运行的时间。但是，这种高级利用率指标是粗略的。它无法告诉有多少SM正在使用中。例如，一个单线程连续运行的内核将获得100%的GPU利用率。

估计SM效率

估计的流多处理器效率。越高越好。内核的这个指标，SM_Eff_K=min(此内核的块/此GPU的SM数量，100%)。这个总体数字是所有内核的SM_Eff_K的总和，按内核的执行持续时间加权，除以“所有步骤时间”。它显示了GPU流多处理器的利用率。虽然它比GPU利用率更细粒度，但它仍然不能说明全部情况。例如，每个块只有一个线程的内核无法充分利用每个SM。

估计实现占用率

对于大多数情况，例如内存带宽受限的内核，更高的值通常意味着更好的性能，特别是当初始值非常低时。占用率的定义（占用率是SM上的活动warp与SM支持的最大活动warp数之比）。内核的理论占用率是此内核的上限占用率，受内核形状，内核使用的资源和GPU计算能力等多种因素限制。内核的估计实现占用率OCC_K=min(内核的线程数/SM数/每个SM的最大线程数，内核的理论占用率），这个总体数字是使用内核的执行持续时间作为权重的所有OCC_K的加权总和。它显示了细粒度的低级GPU利用率。

使用Tensor Cores的内核时间

Tensor Core内核的总GPU时间/所有内核的总GPU时间。值越高越好。Tensor Cores是可用于Volta GPU以及跟高版本的混合精度浮点运算。cuDNN和cuBLAS库包含多个启用Tensor Cores的GPU内核，可用于大多数卷积和GEMM操作。此数字显示GPU上所有内核中的Tensor Cores使用时间比率。

num of workers

指在数据加载过程中用于并行加载数据的工作线程数量。它与PyTorch的DataLoader相关，DataLoader是负责将数据集分成小批量，并将这些批量加载到模型中的。num_workers参数指定了加载数据时使用多少个子进程来并行处理数据。有助于加速数据加载的速度，尤其是在数据集较大且需要进行大量预处理时。

在TensorBoard中查看时，num of workers反映的是训练过程中数据加载部分使用的并行度（即多少个动作线程被用来加载数据）。这个指标有助于了解数据加载的效率和瓶颈，尤其是在数据加载时间较长，是否可以通过增加num_workers来加速。

torch.compile 的编译流程

PyTorch + Inductor + Triton的整个编译流程，特别是triton kernel生成与编译分析这部分的过程。

整体编译流程

  1. 前端捕获（TorchDynamo）

    捕获PyTorch的eager模型，生成一个FX Graph（中间表示IR）

  2. 图优化与分区（AOTAutograd + Decompositions）

    对FX Graph进行算子分解（decomposition）、分区（partition）等预处理。

  3. 后端Lowering（Inductor）

    Inductor接受FX Graph，将算子Lowering成调度后语义（Schedulers）。

    部分算子会映射到Triton-compatible kernels, 由Inductor创建出表示Triton的kernel源码（字符串）

  4. Triton编译器调用

    对于这些Triton kernel源码（通常是Python函数字符串），Triton编译器执行：

     AST（抽象语法树）分析：将kernel函数代码解析为AST。
 语义检查与IR构造：建立Triton的中间表示（SSA IR）。
 后端代码生成：转成GPU可执行代码，比如PTX（NVIDIA）。

torch.compile 常用 Backend 整理（PyTorch 2.x+）

🧠 Backend 一览表

Backend 名称 描述/特点 是否默认 适用场景/建议
"inductor" ⚙️ 默认 backend，基于 Triton 和 C++，性能好、支持广泛 ✅ 默认 推荐首选，支持大部分算子，适用于通用加速优化
"eager" PyTorch 原生模式，不做编译优化 ❌ ✅ Debug 模式；模型不支持编译、想跳过加速或便于排查
"aot_eager" AOT Autograd + Eager，开启 trace 但不做 Inductor 优化 ❌ Debug 早期图 trace 问题；不编译后端，只记录图结构
"nvfuser" 早期 Fusion backend，面向 CUDA（正在逐步弃用） ❌ CUDA 模型上实验使用，不推荐新项目使用
"aot_ts_nvfuser" TorchScript + NVFuser（图优化 + Fusion） ❌ 适用于 TorchScript 路径；性能不一定稳定
"aot_torchxla_traced" 用于 XLA（TPU）后端，例如在 Google Cloud TPU 上 ❌ TPU 训练任务必须使用
"aot_torchxla" 同上，非 traced 版本 ❌ 更可控但 trace 不完整，实验可用
"ipex" Intel Extension for PyTorch，适用于 Intel CPU/GPU ❌ 在 Intel 平台上推荐使用（x86 CPU 加速明显）
"openxla" / "inductor_openxla" 为 OpenXLA 项目构建的 backend，主要用于部署 ❌ 高性能部署场景，开发者可试试 OpenXLA 编译路径（还不太成熟）

🚨 fallback 到 eager 有何区别？

当 torch.compile() 编译失败时（如 GPU 类型不支持），会 fallback 到 eager，这时：

  • 模型依然是用 model_opt = torch.compile(model) 生成的；
  • 实际执行会自动使用原始 Python 模型，不再启用编译加速；
  • 与直接运行 model(x) 无本质区别，仅多一层包装；
  • 可以通过 torch._dynamo.config.suppress_errors = True 开启 fallback。

引言

Docker是什么

  Docker是一种基于容器的虚拟化技术，它能够将应用及其所有依赖打包到一个独立的环境中运行。Docker使应用能够在任何支持Docker的环境中运行，而不会受到底层操系统或硬件的影响。

  Docker由Docker引擎提供支持，它使用容器来运行应用程序。与传统的虚拟机不同，Docker容器直接运行在宿主机的内核之上，因此更轻量，更高效。

Docker主要由以下核心组件组成:

**镜像(Image)**：应用及其环境的只读模板，可以用来创建容器。

**容器(Container)**：基于镜像运行的实例，彼此隔离，类似于轻量级虚拟机。

**仓库(Registry)**：存储和分发镜像的地方，如Docker Hub、阿里云镜像仓库等。

解决了哪些问题

  1. 环境一致性问题
  2. 依赖管理与部署复杂度
  3. 资源占用高、启动慢
  4. 持续集成与交付（CI/CD）困难
  5. 迁移和扩展困难

Docker的原理

容器 vs. 虚拟机

  虚拟机是传统的虚拟化方式。它通过Hypervisor(虚拟机管理程序)在宿主机上运行多个独立的操作系统，每个虚拟机都有自己的内核、文件系统和应用程序。因此，虚拟机的隔离性强，但资源开销大，启动慢。

  Docker容器不需要额外的操作系统，而是共享宿主机的内核，并利用Namespace和Cgroups进行资源隔离和管理。这样容器可以在不影响其他应用的情况下运行，并且占用更少的资源，启动速度极快。

Docker底层技术

  1. Namespace（命名空间）————资源隔离

  Namespace是Linux内核提供的隔离机制，Docker通过Namespace让每个容器由独立的进程、网络、文件系统等。不同容器的进程互不干扰即进程隔离，每个容器有独立的IP和端口即网络隔离，每个容器有自己的根文件系统即文件系统隔离，不同容器的共享内存、信号量等互不干扰即进程间通信隔离。

  1. Cgroups（Control Groups）————资源控制

  Cgroups允许Docker限制容器的CPU、内存、网络、IO资源，防止某个容器占用过多资源，影响主机性能。如限制容器使用1个CPU核心和512MB内存：

  1. UnionFS（联合文件系统）————镜像分层
      UnionFS是Docker镜像的存储机制，它允许多个文件系统层叠加，从而提高存储效率。Docker镜像由多个只读层组成，容器运行时会添加一个可写层。只读层是镜像的基础层（如Ubuntu、Python依赖等）。可写层是运行容器时动态创建，存储用户的修改。通过docker commit可以保存容器的可写层，形成新的镜像。

  1. 看下原理

    1. GPU的硬件模型

      GPU由几十至成百个SM组成，每个SM被分为多个sub-partition，每个sub-partition包含多个cuda core和少数tensor core，除了外挂的显存GPU内部包含L2 cache全局可见，L1 cache，shared memory，它们共用同一个物理内存区域，对一个SM可见，此外每个sub partition还包含寄存器文件等。

    2. GPU的编程模型

      一个kernel对应一个grid，每个grid被分成多个block，每个block包含多个thread，每个block在一个SM上被执行，被划分为多个warp，这些warp被分配给每个sub-partition来执行，每个warp有32线程。

      nvcc将主机端和设备端的代码分别编译、链接生成可执行文件，主机端的代码由CPU执行，设备端的代码由cuda runtime支持在设备端执行。

在 MATLAB 中，你可以使用 “Dependency Analysis”（依赖分析）工具来快速查看函数之间的调用关系。这个工具可以帮助你理解你的代码是如何相互连接的，特别是在大型项目中。

步骤

  1. 打开 MATLAB 并打开你的项目或代码文件。
  2. 在 MATLAB 工具栏上选择 “Editor”（编辑器）标签。
  3. 在编辑器中，打开你想要分析的主函数或代码文件。
  4. 在编辑器中，右键单击函数名称，然后从弹出菜单中选择 “Show Dependencies”（显示依赖关系）。

依赖分析工具

依赖分析工具将在 MATLAB 窗口中显示函数的依赖关系图。这个图示表示了主函数和相关函数之间的调用关系，以及函数之间的依赖关系。
你可以在依赖分析工具中浏览不同函数之间的连接，单击节点以查看详细信息，包括调用图、输入和输出等。
请注意，依赖分析工具可以帮助你可视化函数之间的调用关系，但对于大型项目或复杂代码库，它可能会变得复杂。因此，在代码设计和编写时，良好的代码结构、注释和模块化可以更好地帮助你管理代码的调用关系。
